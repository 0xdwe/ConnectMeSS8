## Architecture Overview

```
┌─────────────────────────────────────────────────────────────────┐
│                    AiUpdateScreen.save()                         │
│                       │                                          │
│              aiUpdate.commit(result)                             │
│                 │          │                                     │
│       memoryStore.save()   appController.applyAiUpdateResult()   │
│       onMemoryWritten()    → batch-writes interaction+connection │
│       bumps memoryEpoch    → state.copyWith(lastAiSummary:...)   │
│                 │          │                                     │
│            ┌────┴──────────┴────┐                                │
│            │ recommendationsProvider  │                           │
│            │   watches memoryEpoch    │                           │
│            │   watches connections    │                           │
│            │   watches interactions   │                           │
│            │   calls rankRecommendations()                       │
│            │   sets lastRecommendationList                       │
│            └────────────────────────┘                            │
│                       │                                          │
│          ┌────────────┼─────────────┐                            │
│    home_tab.dart   recs_screen   AiInsightsCard (via static)     │
└─────────────────────────────────────────────────────────────────┘
```

**The race condition (#117):** `memoryEpochProvider` bumps synchronously during `commit()`, triggering `recommendationsProvider` recomputation. But the interaction write (`applyAiUpdateResult`) goes through Firestore batch → snapshot listener → `AppState.interactions` update, which is async. The provider often recomputes before the new `aiSuggested` interaction appears in `state.interactions`, missing the completion signal.

Current signature:
```dart
List<Recommendation> rankRecommendations({
  required List<Connection> connections,
  required List<CrmInteraction> interactions,
  required Map<String, MemoryDocument> memories,
  required DateTime now,
  List<Recommendation>? previousList,
  DateTime? previousCacheTime,
})
```

Current completion detection (lines 98-148) — the **fallback path**:
1. Iterates `previousList` (when non-null)
2. For each previous entry: skips if already `isCompleted`, skips if still in new list
3. Checks for a new `aiSuggested` interaction with `date >= previousCacheTime`
4. If found, emits a completed `Recommendation` at the original slot position
5. Caps at 1 completed card, returns early

**What needs to change:**
- Add optional `String? lastAiUpdatedContactId` parameter
- Add **fast path** before the fallback: when `lastAiUpdatedContactId` is non-null, check if that contact was in `previousList` and dropped off the new list — if both true, emit a completed card directly (no interaction check needed). Then return early (don't fall through to fallback).
- Fallback path remains unchanged as-is for cases where `lastAiUpdatedContactId` is null or didn't match.

## Implementation Order & Dependencies

#117 MUST be implemented first because #118 depends on `lastAiUpdatedContactId` being consumed by `recommendationsProvider` for the completed state to appear in the `AiInsightsCard` banner.

### #117 Implementation Plan

1. **`AppState`**: Add `lastAiUpdatedContactId` (String?) and `lastAiUpdatedAt` (DateTime?) fields, both defaulting `null`. Update constructor, `copyWith()`, and `seeded()`.
2. **`AppController.applyAiUpdateResult`**: After successful batch commit, set `lastAiUpdatedContactId` and `lastAiUpdatedAt` via `state.copyWith(...)`.
3. **`AppController.clearLastAiUpdate()`**: New method that sets both fields to `null`.
4. **`rankRecommendations`**: Add `String? lastAiUpdatedContactId` optional param. Add fast path: when non-null, check if that contact was in `previousList` and dropped off new list → emit completed card directly. Fallback path (interaction-based) remains unchanged.
5. **`recommendationsProvider`**: Read `lastAiUpdatedContactId` from `AppState`, pass to `rankRecommendations`, then call `clearLastAiUpdate()`.

### #118 Implementation Plan

1. **`_AiInsightsBody`**: Read `recommendationsProvider`, look up current contact's ID. Three states: active banner, completed banner, no banner.
2. **`AiInsightsCard`**: Remove `recommendationReason`, `recommendationInsight`, `recommendationAction` constructor params.
3. **`ContactProfileScreen`**: Remove the three `recommendation*` params.
4. **`connect_me_app.dart`**: Remove the three query-param reads in route builder.
5. **`home_tab.dart` / `recommendations_screen.dart`**: Remove reason/insight/action from URL construction; keep only `topic`.
6. **`_bondEncouragement` function**: Delete (no longer referenced).
7. **Tests**: Update widget tests that reference static params; add new tests for dynamic banner states.

## Known Traps

- `lastRecommendationList` is module-level in `memory_providers.dart` — survives Provider disposal. Tests must reset it in `setUp`/`tearDown`.
- `MockAiUpdate` fakes cannot `implements` final SDK classes (`GenerativeModel`, `GenerateContentResponse`). Use `GeminiGenerateContentFn` function-injection seam.
- Seeded `lastContact` dates use `DateTime.now()` (wall clock). Tests with fake clocks must use explicit old dates.
- Riverpod 3.3.1 `AsyncValue` does NOT have `valueOrNull`. Use `.asData?.value` or `.maybeWhen(data: (...), orElse: () => null)`.
- `AppController` is a god module (27+ methods). Per ADR-0005, cleanup is deferred — do NOT suggest splitting it.
- Snapshot-listener pattern consolidation is deferred per ADR-0006.
