## What to build

Extract expensive query operations from widget `build()` methods into Riverpod derived providers that memoize results and only recompute when their specific data slice changes. This eliminates unnecessary filter/sort/lookup work triggered by unrelated state changes (theme toggle, tab switch).

## Acceptance criteria

## AI Triage Brief

**Category**: enhancement. Performance optimization + architecture improvement. No user-facing behavior changes.

**State**: needs-triage. Requires human review to confirm priority and scope.

**Codebase notes**:
- Current bottleneck: `PeopleTab`, `PlannerTab`, `HomeTab` rebuild on every `AppState` change and re-run expensive queries in `build()`
- `PeopleTab` does O(n log n) filter+sort on every rebuild
- `PlannerTab` queries `selectedEvents` twice per rebuild with `DateUtils.isSameDay()`
- `HomeTab` uses O(n) `firstWhere` for each recommendation card
- `ContactProfileScreen` uses `firstWhere` which throws on missing IDs
- Riverpod providers already in use: `aiUpdateServiceProvider`, `appControllerProvider`

**Implementation notes for the agent**:
- Create `lib/src/state/query_providers.dart`
- Use `Provider.family` for parameterized queries
- Use `.select()` to watch narrow slices: `ref.watch(appControllerProvider.select((s) => s.connections))`
- Map-based lookup for O(1): `{for (var c in connections) c.id: c}[id]`
- Return nullable types for lookups, handle null in widgets
- Test providers with `ProviderContainer` (see `test/state/app_state_test.dart` for pattern)

**Performance impact**:
- Theme toggle: no filter/sort work triggered (currently triggers full People list recompute)
- Tab switch: no query work in inactive tabs (currently all tabs rebuild)
- People search: memoized results, only recomputes when filter changes
- Contact lookup: O(1) instead of O(n)
