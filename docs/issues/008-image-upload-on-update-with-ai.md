## What to build

In the Update with AI screen, allow attaching images alongside the existing free-text and generic file attachments. Image attachments render as small previews; non-image attachments continue to render as labeled chips. Submission flows through the existing mock AI service unchanged.

## Acceptance criteria

- [ ] Update with AI screen has an "Add image" affordance in addition to the existing "Attach" affordance.
- [ ] Images picked via the image affordance appear as image previews (thumbnail) above the submit button.
- [ ] Non-image attachments still render as chips with the file name.
- [ ] Submission still calls `runAiUpdate(...)` with all selected attachments.
- [ ] Submitting a text + image update updates the contact's history (one new interaction visible on the dashboard's History section after returning).
- [ ] A widget test exercises submitting a text-only update and a text + image update.
- [ ] All tests pass.

- Keep tests deterministic: do not invoke the OS picker in widget tests; instead, expose the smallest possible seam to set attachments programmatically (e.g., a `@visibleForTesting` method or a constructor parameter for initial attachments) so tests can verify the rendered state and the post-submit history update without driving the platform picker.
